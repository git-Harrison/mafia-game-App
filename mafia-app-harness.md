# mafia-app — 작업 하네스 (속도 우선)

`mafia-app-context.md`(게임 설계)와 `mafia-app-design-policy.md`(IP)를 함께 읽는다.
이 문서는 **어떻게 빨리 만들지**를 정의한다. 룰·스코프·IP가 이 문서와 충돌하면 저 두 문서가 우선이다.

## 0. 한 줄 원칙

> **병렬 에이전트 + 슬림 테스트 + 자율 결정.**
> 코워크는 짧게 슬라이스 컨텍스트를 던지고, Claude Code 는 가드레일 안에서 자율적으로 결정하며 빠르게 만든다.

## 1. 작업 단위 — 기능 슬라이스 (마일스톤 X)

**디렉터 신뢰 모드 (기본).**

- 한 슬라이스 = 한 인스턴스가 **1~2일 내 완료**할 수 있는 결과물 단위.
- 코워크는 **결과물(what)** 만 정의하고 파일 수·내부 구조는 Claude Code 자율.
- 신규 파일·줄 수에 상한 없음. 컨텍스트 폭주만 안 되면 됨.
- 슬라이스 이름은 `S-<area>-<주제>` 형식(예: `S-server-night-actions`, `S-client-game-screens`).

**미세 모드 (예외).**

- 인터페이스 계약이 잠겨야 하거나 룰 변경처럼 한 줄이 비싼 작업에서만.
- 2~6시간, 최대 4파일 단위로 좁힘.
- 디렉터가 명시적으로 "미세 모드" 신호를 줄 때만 적용.

## 2. 병렬 에이전트 분담

기본 분담(2 인스턴스):

| 인스턴스 | 영역 | 파일 경로 권한 |
|---|---|---|
| **A — Backend** | NestJS 서버, Prisma, WS 게이트웨이, 룰 엔진, 서버 테스트 | `server/**`, `prisma/**`, `docker-compose.yml`, `.env*` |
| **B — Frontend** | Flutter UI, 클라 WS, 수동 검증 | `client/**` |

**충돌 방지 규약:**

- 두 인스턴스가 **동시에 같은 파일을 수정하지 않는다.**
- 인터페이스(WS 이벤트 페이로드)는 변경 전에 코워크가 동기화 → 양쪽에 동일한 갱신본 전달.
- 공유 문서(`mafia-app-context.md` 등)는 한 인스턴스만 편집. 다른 쪽엔 변경분만 전달.

특정 작업이 폭이 좁으면 3번째 인스턴스(예: **C — QA/DevOps**)를 띄워 통합 테스트·CI·디버깅 전담.

## 3. 컨텍스트 전달 표준 (Cowork → Claude Code)

- **길이 200~500자.** 1500자 넘으면 슬라이스가 너무 큰 거다.
- **6항목 고정 구조:**

```
[S-id] <한 줄 목표>

1. 핵심 파일 (1~4개, 경로):
   - server/src/...
2. 인터페이스 계약 (WS 이벤트 / 함수 시그니처):
   - eventName({ ... }) → { ok: true, data? } | { ok: false, error }
3. 자율 결정 OK: <명명, 구조, 테스트 케이스 추가 등>
4. 묻기: <새 의존성, 모델 변경, 룰 변경 등 — 없으면 "없음">
5. 검증: <단위 X개 / 통합 1개 / 수동 시뮬>
6. 금지: <스코프 외 작업>
```

- 슬라이스 5번째 이후로 컨텍스트가 자명하면 1·2번만 보내도 됨.
- design-policy §2 자가검증 의무는 매번 명시할 필요 없음 — Claude Code 가 항시 적용.

## 4. 보고 표준 (Claude Code → Cowork)

- **길이 300자 안.** 폭주하는 보고서는 다음 슬라이스 지연 원인.
- 4항목 고정:

```
✅ 한 것: <파일/변경 요약>
⚠️ 자율 처리한 것: <묻기 영역 아닌데 결정 내린 게 있다면>
❓ 다음 결정 필요: <있으면 옵션 + 추천 / 없으면 "없음, 다음 슬라이스 받음">
🧪 테스트: 단위 X/Y · 통합 X/Y · 수동 OK/NG
```

- 코드 블록·표·중간 결과 dump 금지. 변경 핵심만.

## 5. 결정 권한 가이드

### 자율 결정 (묻지 말고 진행)

- 변수·함수·클래스·파일·폴더 명명
- 라이브러리 마이너 버전 + 패치 버전
- 에러 코드 문구 (design-policy §2 자가검증 후, 기능 서술형)
- 테스트 케이스 추가·강화
- 코드 리팩토링·기술 부채 정리
- 로그 메시지 카피 (자체 작성)
- 코드 주석

### 묻기 (사용자 결정 필수, 슬라이스 정지)

- 새 의존성 추가 (`pnpm add`, `flutter pub add`)
- 데이터 모델 변경 (Prisma 스키마, 메모리 구조체 필드)
- 게임 룰 변경 (룰 엔진 시그니처/동작/룰 자체)
- WS 이벤트 명세 변경 (페이로드 필드, 이벤트명, 에러 코드 신규)
- design-policy §2 hard-no 의심 사항
- 현재 슬라이스 스코프 외 작업
- 환경변수 신규 추가
- 새 파일 5개 이상 한꺼번에 생성
- 인터페이스 계약 변경

### 회색 (보고하면서 진행, 슬라이스 계속)

- 기존 단위 테스트 수정·삭제
- 외부 무료 에셋 추가 (출처·라이선스 기록 필수)
- 새 라이브러리 메이저 버전 호환성 의심
- DB 마이그레이션 (스키마 변경 아닌 데이터 이동·인덱스 추가)

## 6. 테스트 정책 — 슬림

### 강제 100% — 도메인 순수 함수만

- 룰 엔진: `assignRoles`, `tallyVotes`, `resolveNightActions`, `checkWinCondition`
- 향후 추가되는 순수 함수도 동일 기준 적용
- 회귀 99% 를 이 레벨에서 잡는다

### 통합 테스트 — 기능당 1~2개

- WS 이벤트 핸들러, 매니저 메서드, DB 영속화
- 핵심 경로 + 경계 케이스 1개만. 풀 커버리지 X.
- 빠른 환경 (env 로 타이머 1초 단축)

### E2E — 슬라이스 그룹 끝에 1개

- "한 사이클" 시뮬레이션 — 방 생성부터 게임 종료까지
- M-시리즈 끝마다 1개, 슬라이스마다 X

### 수동 검증 — 클라이언트 + 위험 변경 시만

- 시뮬레이터 1~2개에서 직접 클릭
- 영상·스크린샷 의무 아님 (요청 시만)
- Flutter 위젯 테스트 강제 X — 1단계 클라는 단위 0개

### 검증 사이클 비용 vs 회귀 위험

- 도메인 함수 1줄 수정 = 단위 테스트 의무
- 게이트웨이/매니저 1줄 수정 = 기존 통합 테스트 통과면 OK
- 클라 UI 수정 = 시뮬레이터 한 번 돌려서 깨지지 않으면 OK

## 7. 모바일 멀티플레이어 게임 베스트 프랙티스

### 서버

- **Server-authoritative.** 클라는 입력만, 서버가 진실 (적용됨).
- **Phase-based state machine.** 전이는 서버 타이머 만료 또는 모든 행동 완료 중 먼저 도달 (적용됨).
- **In-memory state, persist on terminal events.** 게임 종료 시점에만 Prisma INSERT (적용됨).
- **Reconnection grace 60s.** JWT 같은 토큰 일치 시 자리 보존, 초과 시 사망 처리.
- **Anti-cheat = 매니저 가드.** 게이트웨이는 thin wrapper, 권한·페이즈·상태·입력 검증을 매니저 단일 진입점에 통합.
- **WS handshake auth 는 middleware로.** `handleConnection` async race 회피 (적용됨).
- **Logging 은 gameId 로 trace.** 진행 중 디버깅 시 한 게임만 grep 가능해야.
- **Idempotency.** 같은 socket 두 번 연결 시 옛 연결 disconnect, 새 연결만 활성.
- **Rate limit.** 채팅·재시도 이벤트는 1단계 X, 4단계(스토어 출시) 직전 추가.

### 클라이언트 (Flutter)

- **WS 클라 wrapper.** ack 패턴 `Future<T>` 반환, 자동 재연결, JWT 핸드셰이크 주입 (적용됨).
- **상태 관리: Riverpod 수동.** 코드 생성 X (적용됨, 6단계 확정).
- **화면 전환은 페이즈 이벤트 기반.** `phaseChanged` 수신 → router go.
- **Optimistic UI 금지.** 서버 ack 받기 전엔 UI 변경 X — 서버 권위 일관.
- **Lifecycle.** 백그라운드/포어그라운드 토글 시 WS 재연결 트리거. iOS 백그라운드 가면 WS 끊김 — 푸시 알림은 4단계.
- **에러 UX.** 에러 코드별 한국어 카피 매핑 테이블 (서버는 코드만, 카피는 design-policy §2 자체 작성).
- **딥링크는 4단계.** 방 코드 공유는 1단계엔 텍스트 복사로 충분.

### 모바일 특수

- **개발 네트워크.** iOS sim → `localhost`, Android emu → `10.0.2.2`. env 분기 (적용됨).
- **개발 cleartext.** iOS `NSAllowsLocalNetworking`, Android `network_security_config.xml` 둘 다 (적용됨).
- **빌드 검증.** iOS Simulator 1 + Android 실기기 USB 가 메모리 효율 1순위 (개발 OS 가정: macOS, mafia-app-context.md §7단계).
- **에셋.** Kenney·LPC·CC0 만, 동일 팩이 유명 게임에 채용된 적 있으면 색상·실루엣 변주 (design-policy §3).

## 8. 한 번에 두 인스턴스 굴리는 실전 가이드

### 컨텍스트 동기화

1. 코워크가 슬라이스 한 쌍을 제안 (예: A 는 `S-server-X`, B 는 `S-client-Y`).
2. 두 슬라이스가 같은 인터페이스(WS 이벤트)에 의존하면 인터페이스 페이로드를 두 컨텍스트에 동일하게 명시.
3. 사용자가 인스턴스 A 에 첫 컨텍스트 던지고 → 인스턴스 B 에 두 번째 컨텍스트 던진다.

### 보고 합치기

- 두 인스턴스 보고를 사용자가 코워크에 동시에 붙임.
- 코워크는 두 결과를 검토하고 충돌 (인터페이스 어긋남, 가정 불일치) 있으면 즉시 짚어 차기 슬라이스에서 정렬.

### 충돌 발생 시

- WS 이벤트 페이로드 불일치 → 서버 쪽이 진실, 클라가 맞춤.
- 룰 해석 차이 → `mafia-app-context.md` 가 진실.
- IP/카피 의심 → `mafia-app-design-policy.md` 가 진실.
- 위 셋 다 모호 → 사용자에게 묻는다.

## 9. 1단계 잔여 작업 — 디렉터 신뢰 모드 슬라이스

기준 시점: `S-meta-claude-harness` 완료 (Claude Code 자동 로드 인프라 깔림).

### 인스턴스 A — Backend 잔여

- `S-server-reconnect` — 재접속 grace 60s + `reconnectSnapshot` + 초과 시 사망 처리. 결승선 외이지만 출시 직전 4단계에서 어차피 필요 → 미리 깔아두면 시뮬 검증 중 연결 끊겨도 게임 안 끊김.
- `S-server-e2e-full-cycle` — 풀 사이클 E2E. 결승선 후 회귀 발견 시만 사전 추가.

### 인스턴스 B — Frontend 메인 작업 (1단계 결승선)

- `S-client-game-state` — 게임 이벤트 통합 Notifier + 페이즈별 라우팅 (화면은 placeholder).
- `S-client-game-screens` — **남은 모든 화면 한 큐.** 낮·투표·처형·밤 3페이즈·결과·게임 종료 + 송신 이벤트 + 에러 코드 → 한국어 카피 매핑. 디렉터 신뢰 모드 1~2일치 슬라이스, 파일·줄 수 상한 없음.

### 1단계 결승선 = "한 게임이 처음부터 끝까지 클라이언트에서 동작"

친구 시연·다인원 부하·재접속은 결승선 외. 본인이 시뮬레이터 두 개(iOS sim + Android emu)로 한 판 끝까지 돌려보면 1단계 종료.
